この資料を読むと、何が得られるか

No. 03

Azureならではの装備

企業のAIが乗ってくる「業務基準」の中身

BtoB大企業がAzureを選ぶ最大の理由は、ChatGPT直叩きやその他のAIサービスでは標準では揃えられない、「業務に乗せられるための装備」がAzureには標準で揃っているからです。具体的には3つあります。

① 閉域網で外に出さない構成が組める

Private EndpointとVNet統合を使うと、Foundryやストレージへのトラフィックを企業のVNet内に閉じ込められます。インターネット経由の通信を一切させずに動かせるため、機密情報や個人情報を扱う案件でもそのまま乗せられます。

② Microsoft Entra IDによる認証認可がそのまま使える

社員のアカウントをそのままAIアプリの認証にも使えます。RBACでAzureリソース単位の権限を細かく設定でき、サービスプリンシパルやマネージドIDを使えばアプリ側からのトークン取得も自動化できます。社内ID基盤との統合がデフォルトで効くのがAzureの強みです。

③ 監視・コスト管理が一気通貫で動く

Azure Monitorでメトリック収集、メトリックアラートで異常検知、Action Groupでメール／Logic Apps／Webhook等への通知振り分けが標準で組めます。Cost Managementと組み合わせれば、トークン使用量の急増もコスト超過もリアルタイムで把握できます。

この装備があるかどうかが「業務で使えるAI」の分かれ目 PoCで終わるAIと、実際にお客様の業務に乗るAIの差は、モデルの賢さではなくこの3つの装備が揃っているかどうかです。Azureはこの3つが「最初から揃っている」ことに価値があります。
本資料の家族用AIエージェントでも実装しています 後半で紹介する家族用AIには、Entra ID認証（Static Web Apps Easy Auth）／メトリックアラート（トークン使用量超過の検知）／Logic AppによるLINE通知 を組み込んでいます。最小スケールでも同じ装備が一通り使えます。
No. 04

BtoB業務AIで使われている機能・ユースケース

大企業のAI予算は、ここに流れています

実際にBtoBの現場でよく見るAI活用は、大きく3つのパターンに分かれます。どれもMicrosoft Foundryの範囲内で実装可能です。

① 社内ドキュメント検索（RAG）

社内規程・マニュアル・過去案件資料などをナレッジ化し、自然言語で問い合わせるパターン。Foundry Agent Service ＋ Foundry IQ、もしくはFoundry ＋ Azure AI Searchで組みます。BtoB案件のAIアプリで最も多いユースケースです。

② エージェント自動処理

社内システムや外部APIをツールとして繋ぎ、AIが判断しながら一連の業務を自動で処理するパターン。Foundry Agent ServiceのTool Catalog（1,400以上のツール接続が可能）を使い、複数エージェントのオーケストレーションも組めます。問い合わせ対応・申請ワークフロー・データ調査などで使われます。

③ 文書解析・読み解き

請求書・契約書・申請書などの構造データ抽出、画像や動画の内容理解。Document Intelligenceで帳票を構造化、Content Understandingでマルチモーダル文書（図表混在のPDFや動画）を読み解きます。バックオフィス自動化の主戦場です。

繰り返しお伝えしたいこと これらは大企業のBtoB案件で実際に予算が動いている領域です。同じ「AIに触れるエンジニア」でも、Azureでこれらを組める人材は希少で、単価帯が一段上を狙える構造になっています。今このスキルを身につけておく価値は、十分に投資対効果に見合うはずです。
No. 05

アプリ構成パターン ─ どう組み合わせるか

バックエンド／フロントエンドの選び方と、AI部品の組み合わせ方

AIアプリは、フロントエンドとバックエンドのホスティング選択肢、そしてAI部品（プロンプト・インデックス・エージェント）の組み合わせ次第で、構成が大きく変わります。ユースケースに合わせて使い分けるのが基本です。

バックエンドの選択肢

AS
App Service
常時稼働Webアプリ向け
FN
Azure Functions
イベント駆動・短時間処理
CA
Container Apps
コンテナベースのスケーリング

App Serviceは社内向けの常時稼働Webアプリに、FunctionsはSWAと組み合わせたサーバーレスAPIや、ファイル投入をトリガーにした処理に向きます。Container AppsはDocker前提のチームや、より柔軟なスケーリングが必要な場合に選ばれます。

フロントエンドの選択肢

SWA
Static Web Apps
Reactなどの静的SPA向け
AS
App Service (Web)
サーバーサイドレンダリング向け

軽量なフロントエンドや個人開発・小規模アプリはStatic Web Apps。Easy Auth（Entra ID連携）が標準装備で、認証付きアプリが追加実装ゼロで作れます。サーバーサイドレンダリング（Next.jsのSSRなど）が必要な大規模アプリはApp Serviceを選びます。

AI部品の組み合わせパターン

代表的な組み合わせ
  • シンプルなRAGアプリ：SWA → Functions → Foundry Agent + File Search（最小構成）
  • 規模を持つRAGアプリ：App Service → Foundry Agent + Azure AI Search（独立検索）
  • エージェント自動処理：Container Apps → Foundry Agent Service（複数エージェント・Tool Catalog経由で外部API連携）
  • 文書処理パイプライン：Blob Storage → Functions → Document Intelligence → Foundry Models（要約・分類）
本資料の家族用AIエージェントは「最小構成」を選択 後半で紹介する実例は Static Web Apps（フロント／認証） + Foundry Agent Service（バック／File Search） の最小スタックです。バックエンドコードはSWA Managed Functions（Node.js）でプロキシだけ書いている形になります。
No. 06

自分で試す最小構成

無料〜低価格で動かせる、入口の最短ルート

Azure AIは「触ってみないと分からない」のが本音です。月数百円規模で動かせる最小スタックを以下にまとめました。所要時間は2〜3時間ほどです。

最小スタック構築の流れ

  1. Microsoft Foundryプロジェクトを作成（Azure Portal → Microsoft Foundry → 新規プロジェクト）
  2. モデルをデプロイ（gpt-5-mini など。Global Standard・TPM 10K で十分）
  3. エージェントを作成（Foundry Agent Service。指示文と利用ツールを設定）
  4. ナレッジを登録（File Searchに必要なPDFをアップロード）
  5. Static Web Appsをデプロイ（GitHub連携 → Reactフロント＋Functionsバック）
  6. SWA Easy Authを有効化（Microsoftアカウントによる認証）
  7. サービスプリンシパルを作成してSWA環境変数に設定（Foundry認証用）

手順の詳細はFoundry公式ドキュメント（learn.microsoft.com/azure/foundry/）と、後述するサンプルリポジトリのREADMEに記載しています。各ステップで詰まりやすい認証周りの落とし穴も併せて書いてあります。

ハマりがちなポイント Foundry Agent ServiceはAPIキー認証に対応していません。SWA FunctionsからFoundryを呼ぶときは、サービスプリンシパル（クライアントID＋シークレット）でEntra IDからトークンを取得し、Bearerヘッダで渡す必要があります。これを知らないと数時間ハマります（私も最初ハマりました）。
料金の目安 Static Web Apps Free（無料）／Foundry Models（gpt-5-mini）：使った分だけ／File Search（Foundry IQ）：使った分だけ／Logic Apps Consumption：月4,000アクションまで無料。家族数人での利用なら月10円〜数十円に収まります。

No. 08

私のキャッチアップ術

毎日の流れと、業務にどう活かしていくか

Azure AIは進化が速く、2025〜2026年だけでも大きな改名や統合が起こっています。私は次の4ステップで「速さ」と「正確さ」を両立させています。

STEP 1：GA／Previewのアナウンスを受け取る

Microsoft公式のAzure UpdatesとMicrosoft 365 RoadmapのRSSをチェック。X（旧Twitter）でMicrosoft関係者やAzure系のコミュニティをフォローしておくと、GAタイミングが流れてきます。

STEP 2：Microsoft Learnで概要を読む

新サービス・新機能はlearn.microsoft.comに必ずドキュメントが立ちます。「What is 〜」「Quickstart」「Architecture」の3つを順に読めば、何ができるか・どう使うか・どこに位置付けられるかが10分で掴めます。

STEP 3：料金を確認する

新サービスはまずAzure料金計算ツールか公式のPricingページで料金を確認します。これをやっておかないと「触ったら高そう」で手が止まりがちなので、最初に「最小構成だと月いくら」を把握するのが大事です。

STEP 4：Azureポータルで実際に触る

最後に必ず手で動かします。個人のAzureサブスクリプションか、会社の検証環境を借りて、最小構成でデプロイ→画面確認→削除、までやります。読むだけでは絶対に身につかないので、ここまでやってはじめて「使える」レベルになります。

前作のキャッチアップ用Webサイト集と合わせて読むと効率が出ます Azureに関する情報源・公式ドキュメント・コミュニティの一覧は、別資料「しゃま愛用／実務のためのAzure関連Webサイト集」にまとめています。LINE登録時の特典として配布していますので、合わせてご活用ください。

手を動かした分だけ、
Azureは身近になる。

本資料では、構成と考え方の紹介に絞りました。最近は生成AIのおかげで、こうした検証はとてもしやすくなっています。それでも、実際にAzureポータルで手を動かして、認証で詰まったり、料金が想定外に出たり、エージェントが意図しない返答をしたりする体験には、置き換えられない価値があります。

生成AIにIaCコードを書かせたり、構成を提案させたり、楽をする手段はいくらでもありますが、地道に画面で動かしてみる作業こそが、案件で使えるレベルへの一番の近道です。本資料はあえて構成のみの紹介としました。公式ドキュメントを参照しながら、ご自身の業務に当てはめて、ぜひ手を動かしてみてください。
